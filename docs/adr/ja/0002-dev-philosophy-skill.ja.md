# 0002: ユーザーの開発哲学を第二のスキルとしてパッケージングする

(English: [0002-dev-philosophy-skill.md](../en/0002-dev-philosophy-skill.md))

## Context(背景)

これまでプラグインにはスキル1つ(`fable-coding`、コーディングタスクの規律)と認証情報 deny フックが入っていた。ユーザーはこれとは別に開発*哲学* — 人間中心・Foolproof 設計、自動化の境界線(ヒューマンインザループ、GAS/Discord など外部信頼サービス優先)、プロンプトではなく仕組みによるセキュリティ、案件特性による言語選定、仕様・ADR 相当の内容のコードコメント化、MIT License 標準 — を持っており、マシンごとの memory や CLAUDE.md ではなく、`fable-coding` をインストールしたどの環境でも自動適用されるようプラグインに同梱したかった。

## Decision(決定)

第二のスキル `skills/dev-philosophy/SKILL.md` を追加した(v1.3.0、1.3.1–1.3.2 で拡張)。本文は日本語(ユーザー自身の文言を軽く構造化)。フロントマットの `description` は英語でトリガー重視の書き方にした — スキルの自動マッチングは description で判定されるため。発動対象: システム/アーキテクチャ/自動化/UI・UX の設計、新規プロジェクト、言語選定。構成: 1. 人間中心設計 (Human-Centered & Foolproof)、2. 自動化の境界線、3. セキュリティ哲学、4. 開発・アーキテクチャ方針(言語マップ: DB 操作は Rust、AI・DS・Discord Bot は Python、組み込みは C/C++、Web は TypeScript。MariaDB/Proxmox、MIT)、5. Claude への行動指示(承認フロー、外部サービス優先の提案)。

## Alternatives rejected(却下した代替案)

- **Auto-memory / CLAUDE.md** — マシン単位・プロジェクト単位であり、プラグインと一緒に持ち運べない。ユーザーはプロンプトファイルの分散より仕組みを明確に好む。
- **`fable-coding` スキルへの統合** — トリガーの範囲が違う。あちらはコーディングタスクで発動するが、哲学はコードが存在する前の設計・アーキテクチャ提案でも発動する必要がある。

## Consequences(帰結)

- プラグインを入れた環境なら、設計系タスクで哲学が追加設定ゼロで適用される。更新は SKILL.md の編集とバージョンアップだけで済む。
- 哲学の「仕様・ADR 相当の内容をコードコメントとして残す」ルールは、`fable-coding`/ponytail の「コメント最小限」スタイルを意図的に上書きする — 明示的なユーザールールはスタイルのデフォルトに勝つ。
- 「人間中心設計」をセクション1に置いたのは、既存のヒューマンインザループや物理的制限がその具体化として読めるため。
- 上限: スキルの発動はモデル判断(description マッチング)であり、対象ターンで必ず発動する保証はない。確実性が必要ならフックになるが、この内容にそこまでは不要。
